# Крикет новинарски агрегатор

Автоматизирана система, която следи крикет новините от няколко източника, открива
кога различни сайтове отразяват една и съща новина (семантична прилика на заглавия),
обединява съвпадащите материали в нов текст чрез езиков модел и ги публикува като
чернови на официалния сайт.

Този файл е **единният източник на истина** за проекта. Обновява се при всяка
завършена фаза.

---

## Как работи (накратко)

Едно насрочено изпълнение минава през цялата верига последователно и приключва:

```
RSS емисия (нативна, където минава) или Google News посредник
        │  (заглавие + URL)
        ▼
   дедупликация (embeddings + косинусова прилика, праг 0.8)
        │  (нови съвпадащи двойки между източници)
        ▼
   извличане на телата (trafilatura; fetch-услуга за блокираните сайтове)
        │
        ▼
   обединяване в нов материал (LLM)
        │
        ▼
   публикуване като чернова → API на сайта
```

Изпълнението живее в **GitHub Actions** (по разписание), а локалната машина служи
само за разработка.

## Взети решения

| Тема | Решение |
|---|---|
| Хостинг / изпълнение | GitHub Actions по разписание (cron ~15 мин), „тънък“ конвейер |
| Видимост на хранилището | **Публично** — безплатни неограничени Actions за честия график; тайните остават скрити чрез `.gitignore` + Actions Secrets |
| Стратегия за скрейпване | Нативен RSS където минава (ESPN, Wisden, HT) → Google News посредник за останалите → `trafilatura` за телата, **Zyte API** резерв за блокираните сайтове |
| Дисциплина за цената | Платените методи само когато безплатният се провали — не на всеки цикъл |
| Извличане на тела (резерв) | **Zyte API** (`httpResponseBody`, анти-бот, без браузърски рендер) — само за News18, NDTV, ESPNcricinfo, Indian Express (плейн заявката при тях е 403); работещите сайтове остават на безплатния `trafilatura` път |
| Състояние / дедупликация | SQLite (заменя крехкото сравняване на JSON файлове) |
| Устойчивост | Всеки източник в собствен `try/except` — счупен източник не сваля конвейера |
| Публикуване | Чрез защитен API endpoint на официалния сайт, със статус **чернова** |
| Обединяване | **OpenAI `gpt-4.1-mini`** — нов оригинален материал (свежо заглавие + синтез на фактите със свои думи), статус **чернова**; само за двойки с налични тела на двете статии |
| Embeddings (дедупликация) | **OpenAI `text-embedding-3-small`** (API) — embed-ват се само новите заглавия, по едно пакетно извикване на цикъл; косинусова прилика, праг 0.8 |

## Висящи решения

- _Няма._ Въпросът „локални (MiniLM) vs API embeddings“ е решен на Фаза 4 в полза
  на OpenAI `text-embedding-3-small` (леко изпълнение в Actions, без модел за теглене).

## Източници

Пълната, потвърдена карта е в **`SOURCES.md`**. Накратко: **10 използваеми източника** —
3 през нативна RSS емисия (ESPNcricinfo, Wisden, Hindustan Times) и 7 през Google News
посредник (Times of India, Indian Express, News18, NDTV, Cricket World, Cricbuzz, ICC).
Crex отпадна (остаряло съдържание). Cricket.com е неактивен.

Важно следствие за Фаза 3: телата на статиите от Indian Express, News18 и NDTV вероятно
ще изискват fetch-услуга (същият 403, който видяхме на нативните им емисии).

## Технологичен стек

- **Език:** Python (последна стабилна 3.x)
- **Събиране:** `feedparser` (RSS + Google News посредник)
- **Извличане на тела:** `trafilatura` (основен); **Zyte API** резерв за анти-бот блокираните
- **Дедупликация:** embeddings (OpenAI `text-embedding-3-small`) + косинусова прилика (numpy), праг 0.8
- **Съхранение:** SQLite
- **Обединяване:** OpenAI `gpt-4.1-mini` (нов оригинален материал от двата източника)
- **Разписание:** GitHub Actions (cron)
