# Крикет новинарски агрегатор

Автоматизирана система, която следи крикет новините от няколко източника, открива
кога различни сайтове отразяват една и съща новина (семантична прилика на заглавия),
обединява съвпадащите материали в нов текст чрез езиков модел и ги публикува като
чернови на официалния сайт.

Този файл е **единният източник на истина** за проекта. Обновява се при всяка
завършена фаза.

---

## Как работи (накратко)

Едно насрочено изпълнение минава през цялата верига последователно и приключва:

```
RSS емисия (нативна, където минава) или Google News посредник
        │  (заглавие + URL)
        ▼
   дедупликация (embeddings + косинусова прилика, праг 0.8)
        │  (нови съвпадащи двойки между източници)
        ▼
   извличане на телата (trafilatura; fetch-услуга за блокираните сайтове)
        │
        ▼
   обединяване в нов материал (LLM, gpt-5.4-mini)
        │  + дедупликация на събития (12h) — дубликатите се прескачат
        ▼
   ревю-имейл до човек (SMTP/SSL 465) → одобрение
        │
        ▼
   (по-късно) публикуване като чернова → API на сайта
```

Изпълнението живее в **GitHub Actions** (по разписание), а локалната машина служи
само за разработка.

## Взети решения

| Тема | Решение |
|---|---|
| Хостинг / изпълнение | GitHub Actions по разписание (cron ~15 мин), „тънък“ конвейер |
| Видимост на хранилището | **Публично** — безплатни неограничени Actions за честия график; тайните остават скрити чрез `.gitignore` + Actions Secrets |
| Стратегия за скрейпване | Нативен RSS където минава (ESPN, Wisden, HT) → Google News посредник за останалите → `trafilatura` за телата, **Zyte API** резерв за блокираните сайтове |
| Дисциплина за цената | Платените методи само когато безплатният се провали — не на всеки цикъл |
| Извличане на тела (резерв) | **Zyte API** (`httpResponseBody`, анти-бот, без браузърски рендер) — само за News18, NDTV, ESPNcricinfo, Indian Express (плейн заявката при тях е 403); работещите сайтове остават на безплатния `trafilatura` път |
| Състояние / дедупликация | SQLite (заменя крехкото сравняване на JSON файлове) |
| Персистенция на базата | **GitHub Actions cache** (не git commit) — `data/cricket.db` се restore-ва/save-ва между пусканията с rolling ключ `cricket-state-*`. Началното състояние идва от малкия committnat `data/seed.db` при cache miss. (Причина: commit-back-ът чупеше при > 100 MB — виж root-cause бележката.) |
| Размер на базата | Всяко пускане: занулява embedding-и > 3 дни (основният размер), трие редове > 14 дни, ненужни на дедупа, и прави **VACUUM**. Държи файла малък (~единици/десетки MB), не расте неограничено |
| Архив на новините | Отделен **`data/archive.db`** (append-only, dedup по url) — source, headline, url, published, first_seen, тяло където има + обединените материали. Персистира се като **Actions артефакт** (`cricket-archive`, ретенция 90 дни). НЕ се commit-ва и НИКОГА не се чете от конвейера (само запис). За бъдещ match-prediction проект |
| Устойчивост | Всеки източник в собствен `try/except` — счупен източник не сваля конвейера |
| Устойчивост при преходни грешки (Фаза 8) | Общ retry (`src/retry.py`): до 3 опита, експоненциален backoff с jitter (~1s, 2s, 4s), логван. Повтаря САМО преходни грешки (connection/timeout, HTTP 429/5xx, OpenAI APIConnection/APITimeout/RateLimit/InternalServer); проваля се веднага при постоянни (HTTP 4xx освен 429, OpenAI Authentication/BadRequest, SMTP Auth). Обвити: Google News/RSS заявки, OpenAI embeddings+combine, Zyte, SMTP. Retry-ите са вътре в per-source try/except и не заобикалят предпазните тавани. Google News заявки се разреждат с ~1.5s пауза, за да не приличат на бот burst (503) |
| Публикуване | Чрез защитен API endpoint на официалния сайт, със статус **чернова** |
| Обединяване | **OpenAI `gpt-5.4-mini`** със синтез-промпта v2 — нов оригинален материал (свежо заглавие + чисто тяло + EDITOR NOTES), статус **чернова**; само за двойки с налични тела на двете статии. Преди заявка телата минават през `clean_body()` (маха bylines, агенции, „Updated:“ редове). След генериране — детерминистична пост-обработка: маха em dash от тялото, нормализира EDITOR NOTES |
| Embeddings (дедупликация) | **OpenAI `text-embedding-3-small`** (API) — embed-ват се само новите заглавия, по едно пакетно извикване на цикъл; косинусова прилика, праг 0.8 |
| Прозорци (съвпадение / събитие) | **12 часа** — и за кръстосаното съвпадение (Фаза 4), и за дедупликацията на събития (Фаза 6) |
| Дедупликация на събития (Фаза 6) | Преди обединяване: ако коя да е статия прилича (косинус ≥ 0.8) на статия от вече публикувано събитие в последните 12h → прескача се (без обединяване, без разход). Маркира се `combine_status` ('combined' / 'duplicate') |
| Ревю-доставка (Фаза 6) | Имейл до ревюър през **SMTP над SSL, порт 465** (`smtplib.SMTP_SSL`, не STARTTLS): тема = заглавието, тяло = статия + EDITOR NOTES + двата източника. Идемпотентно (`emailed_at`). Контрол на пращането: `--limit N` (праща най-много N този път; `--limit 1` = безопасен единичен тест) и изричен `--dry-run`. Празни SMTP променливи → dry-run („would send“) |

## Технологичен стек

- **Език:** Python (последна стабилна 3.x)
- **Събиране:** `feedparser` (RSS + Google News посредник)
- **Извличане на тела:** `trafilatura` (основен); **Zyte API** резерв за анти-бот блокираните
- **Дедупликация:** embeddings (OpenAI `text-embedding-3-small`) + косинусова прилика (numpy), праг 0.8
- **Съхранение:** SQLite
- **Обединяване:** OpenAI `gpt-5.4-mini` (нов оригинален материал + EDITOR NOTES; пост-обработка маха em dash)
- **Дедупликация на събития:** embeddings + косинус ≥ 0.8, прозорец 12h
- **Ревю-доставка:** `smtplib.SMTP_SSL` (порт 465) — имейл до човек за одобрение
- **Разписание:** GitHub Actions (cron)

### Root-cause: многодневният срив (2026-07-13)

`data/cricket.db` порасна до **100.70 MB** (embedding вектори: 1536 float-а/заглавие като JSON).
GitHub отхвърли push-а (лимит 100 MB) → стъпката „Persist DB state (commit back)“ падаше → **всеки
run падаше**. Понеже commit-back-ът не минаваше, всеки следващ run дърпаше **стара** база и
преобработваше вече-изпратени материали → **дублиращи имейли** (напр. „Shapoor Zadran dies“ в 15:00,
18:09, 20:27). Алармата „ран зелена“, но не пристигаше, защото темата беше на кирилица (спам).

**Поправка:** персистенция през Actions cache (не git) → без 100 MB лимит; подрязване + VACUUM всеки
цикъл → базата не расте; архив на новините през артефакт; английска тема на алармата. Понеже base-ът
вече наистина оцелява между run-овете, повторно пускане не праща дубликати и не харчи API за вече

### Как да изтегля архива на новините

Архивът `data/archive.db` (за бъдещия match-prediction проект) се качва като **артефакт** — веднъж
дневно (първото пускане след 00:00 UTC) и при всяко ръчно пускане. За да го вземеш:

1. GitHub → таб **Actions** → workflow **„Cricket news pipeline“**.
2. Отвори последното (или ръчно пуснато) изпълнение.
3. Долу в секция **„Artifacts“** → кликни **`cricket-archive`** (сваля `archive.db`).
   За най-свеж архив: **„Run workflow“** (ръчно пускане) и после свали артефакта от този run.

⚠️ **Важно:**
- **Артефактите изтичат** (ретенция 90 дни — максимумът на free плана). **Сваляй архива периодично**,
  иначе се губи. Пълната история досега е и локално в `data/archive.db`.
- **Ако архивът порасне голям дългосрочно**, той трябва да отиде в истинско object storage
  (**S3 / Cloudflare R2 / Backblaze B2**), а не да стои в Actions артефакти.

**Фаза 6 завършена (ревю-имейл, human-in-the-loop).** Обединяването вече е през **`gpt-5.4-mini`**
(`src/combine.py`) със синтез-промпта v2; преди заявка телата минават през `clean_body()` (маха bylines,
агенции, „Updated:“ редове). Добавена е **дедупликация на събития**: ако статия прилича (косинус ≥ 0.8) на
вече публикувано събитие в последните **12h**, двойката се прескача без разход (`combine_status`). След
генериране — детерминистична пост-обработка: em dash се маха от тялото, EDITOR NOTES се нормализира.
Новите материали се пращат за ревю по имейл (`src/notify.py`) през **`smtplib.SMTP_SSL`, порт 465** (тема =
заглавие; тяло = статия + EDITOR NOTES + двата източника); идемпотентно по `emailed_at`, а при празни SMTP
променливи прави dry-run. Тестово изпълнение: 42 двойки → **27 обединени, 13 дубликати, 2 без тяло, 27
would-email**. Прозорецът за съвпадение (Фаза 4) е свит на 12h.

**Фаза 5 завършена.** `src/combine.py` обединява всяка готова двойка (с тела на двете статии) в

със свои думи. Резултатът се пази в `combined_articles` (статус `draft`), идемпотентно по `match_id`.
Двойките без тяло се прескачат без да чупят цикъла; заявка тръгва само за нова, готова двойка
